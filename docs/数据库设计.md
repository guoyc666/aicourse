# 数据库设计文档

## 1. 数据库环境说明

1. 所采用的数据库系统与配套工具

- 关系型数据库：MySQL 8.0
- 编程工具：Python + SQLAlchemy ORM，迁移工具使用 Alembic。

2. 详细配置示例

- 环境变量（位于 backend/.env 或容器环境）：

  - DATABASE_URL=mysql+pymysql://<user>:<password>@<host>:3306/<dbname>?charset=utf8mb4

- MySQL 推荐配置（production）建议：

  - character_set_server = utf8mb4
  - collation_server = utf8mb4_general_ci
  - innodb_file_per_table = ON
  - innodb_buffer_pool_size = 1G (按内存调整)
  - max_connections = 200
  - time_zone = '+08:00'

- SQLAlchemy/Engine 建议参数：

  - pool_size: 10-20（生产）
  - pool_recycle: 1800
  - pool_pre_ping=True
  - 使用 `mysql+pymysql` 驱动

- 备份与恢复：
  - 周期性全备（mysqldump / Percona XtraBackup），每日增量备份，定期演练恢复流程。

## 2. 数据库命名规则

- 表名：使用小写下划线（snake_case），多为复数或语义清晰名，例如 `users`, `roles`, `file_resources`。
- 主键：使用 `id` 或在已命名资源中使用 `<entity>_id`（例如 `question_id`、`file_id`）。主键类型按实体选择整数自增或 UUID 字符串（长度 36）。
- 外键：使用 `<referenced_table>_id` 或明确的列名，例如 `created_by_id`, `knowledge_id`。
- 列名：采用小写下划线，语义化命名，如 `created_at`, `updated_at`, `is_active`。
- 索引：常查询列应建立索引并使用 `*_idx` 命名规范（在 SQL DDL 中显式命名可选）。
- 枚举：使用数据库 ENUM 或在应用层用字符串字段限制（本项目在 SQLAlchemy 中使用 Enum 类型）。

以下为项目主要实体及其关系的文字形式 ERD 概述

- User (users) 1 --- _ UserRole (user_roles) _ --- 1 Role (roles)
- Role (roles) 1 --- _ RolePermission (role_permissions) _ --- 1 Permission (permissions)
- User 1 --- \* Task (tasks) （创建者）
- Task 1 --- \* AssignedTask (assigned_tasks) --- 1 Student(User)
- Task 1 --- \* TaskSubmission (task_submissions) --- 1 Student(User)
- Topic 1 --- \* Reply (replies)
- FileResource 存储上传资源，关联上传者 User
- Question（题库）可关联 KnowledgePoint（knowledge_points）
- KnowledgePoint 与 Resource 通过 KnowledgeResourceLink 建立多对多
- Mastery 表记录 Student(User) 与 KnowledgePoint 的掌握度（复合主键 student_id, knowledge_id）
- Conversation (conversations, models\_/ai_assistant.py) 1 --- \* Message (messages)

## 5. 安全性设计

### 5.1 防止用户直接操作数据库的方法

- 所有用户通过应用（FastAPI）访问数据库，数据库账号仅由后端服务使用；不直接对外开放 DB 管理端口（如 3306）到公网。
- 生产环境通过私有网络或 VPN、内网子网隔离数据库访问。外部管理仅允许受控跳板机或运维 VPN。
- 使用只读/只写分离角色账号，限制运维或监控账号的权限。

### 5.2 用户帐号密码的加密方法

- 密码使用 bcrypt（或 PBKDF2/Argon2）进行单向哈希并存储哈希值（字段 `hashed_password`），并且后端不在日志或接口返回中记录密码明文。
- 通过 HTTPS/TLS 保护网页与 API 通信，防止明文传输。

### 5.3 角色与权限

- 在应用层使用 RBAC（roles / permissions / user_roles / role_permissions 表）来控制功能访问。
- 数据库级别采用数据库账号最小权限原则：
  - 应用账号：拥有对应用需要表的 CRUD 权限；
  - 备份账号：有备份和读权限；
  - DBA 账号：仅在必要时使用，并受操作审计。

| 角色            | 可以访问的表与列                                                                                                                                                                                                                                                       | 操作权限                        |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------- |
| 管理员（admin） | users、roles、permissions、user_roles、role_permissions、file_resources、question、tasks、assigned_tasks、task_submissions、topics、replies、knowledge_points、resources、knowledge_resource_links、mastery、learning_record、question_record、conversations、messages | 全权限（慎用）                  |
| 教师（teacher） | question（创建/编辑/发布）、tasks（创建/分配）、assigned_tasks、task_submissions、topics、replies、file_resources（上传/管理）                                                                                                                                         | 增删改查（受审核/发布流程约束） |
| 学生（student） | assigned_tasks、task_submissions（创建/更新）、question_record（创建）、topics、replies                                                                                                                                                                                | 创建 / 读取 / 更新（受限）      |

## 6. 优化

### 6.1 分析与优先级

| 优先级 | 优化对象（目标）          | 措施                                                                                           |
| ------ | ------------------------- | ---------------------------------------------------------------------------------------------- |
| 高     | 向量检索上下文 / 检索延迟 | 将向量索引（Chroma / FAISS）放在独立服务；暂存常用检索结果到 Redis 缓存。                      |
| 高     | 题库 / 答题查询速度       | 对 `question` 的 `knowledge_id` 做必要的全文索引或将 JSON 字段拆表；常用题集缓存到 Redis。     |
| 中     | 大量日志 / 历史记录表查询 | 使用分区或按时间分表（如 `question_record`、`task_submissions`）以提高查询速度并降低单表膨胀。 |
| 低     | 统计计算                  | 批处理离线计算并写回汇总表，避免实时复杂联表。                                                 |

### 6.2 具体优化措施

- 索引策略：为常用过滤/连接的列添加索引（users.username, file_resources.file_id, question.question_id, knowledge_resource_links.knowledge_id 等）。
- 读写分离：高并发场景下考虑主从复制，读请求导向只读从库。
- 缓存：使用 Redis 缓存热点查询与会话数据，减少数据库压力。
- 批量/异步：嵌入计算（embedding）、题目生成、日志写入等采用后台任务（Celery / RQ）异步处理。
- 表设计折中：对于查询性能要求极高的场景可适度反规范化（例如题库的选项单独列或选项表），并在写入时做好一致性控制。

## 7. 数据库管理与维护说明

- 数据库迁移：使用 Alembic 管理版本化迁移脚本。开发分支在合并前运行迁移并验证回滚。
- 备份策略：每日全备 + 每日/每小时增量（按业务量）。备份保留策略按团队规定（例如 30 天内可恢复）。
- 恢复演练：每季度至少一次演练恢复流程，验证备份可用性。
- 监控与告警：通过 Prometheus + Grafana 监控连接数、慢查询、磁盘使用、QPS/延迟，配置告警阈值。
- 安全审计：对敏感操作（如用户删除、权限变更、关键表 DDL）记录审计日志并保留审计记录。
- 日常维护：

  - 优化慢查询并添加索引；
  - 清理过期临时数据（如 temp_uploads、测试数据）；
  - 检查表碎片并运行 OPTIMIZE TABLE（MySQL）或相应维护操作；
  - 更新依赖（驱动/库）并在测试环境验证后部署。

- 运维脚本/工具：
  - 备份脚本（mysqldump / xtrabackup），压缩并上传到对象存储；
  - 数据库迁移（alembic upgrade head）；
  - 数据导出/导入工具（CSV/JSON/QTI），并提供差错报告。

### 环境说明

- 系统：Neo4j（5.x LTS 版本）。
- 部署方式：Docker Compose（独立容器）。
- 连接：Bolt URI，示例：bolt://neo4j-host:7687。
- 认证：使用环境变量管理账号/密码（NEO4J_AUTH 或 secrets），禁止在仓库中提交明文凭据。
- 资源建议：内存与 pagecache 按实例大小配置，示例：--env NEO4J_dbms_memory_pagecache_size=1G，堆内存根据负载调整（-Xmx/-Xms）。
- 管理/监控：启用指标导出（Prometheus 插件）与定期备份（neo4j-admin dump/restore）。

### 数据结构（节点、关系与属性）

说明：下列为知识图谱常用节点与关系类型，按系统需求可扩展。

节点（Labels）示例：

- KnowledgePoint

  - id: string (主键，建议短码或 UUID，例如 'kp-001')
  - name: string
  - description: text
  - category: string
  - source: string (来源资源标识)
  - created_at: datetime

- Resource

  - id: string
  - title: string
  - url: string
  - mime_type: string
  - created_at: datetime

- Concept / Topic（可选，用于更细粒度概念分层）

  - id, name, description, tags

- Student（用于存储与图谱直接关联的学习实体，可选）
  - id (与关系型 users.id 对应)
  - name

关系（Relationship types）示例：

- (a:KnowledgePoint)-[:PRECEDES {weight:float}]->(b:KnowledgePoint)

  - 表示先修/依赖关系，weight 可表示强度/必需程度

- (kp:KnowledgePoint)-[:HAS_RESOURCE {role:string}]->(r:Resource)

  - 表示知识点与资源的关联，role 可为 'lecture','exercise' 等

- (kp1)-[:RELATED_TO {score:float}]->(kp2)

  - 表示语义相似或相关度，score 存放相似度

- (student)-[:MASTERED {level:float, updated_at:datetime}]->(kp)
  - 学生对知识点的掌握程度，可用于个性化推荐

属性与索引建议：

- 在节点常用查询字段上建立索引/唯一约束：
  - CREATE CONSTRAINT ON (k:KnowledgePoint) ASSERT k.id IS UNIQUE;（Neo4j 4.x/5.x 语法视版本调整）
  - CREATE INDEX FOR (r:Resource) ON (r.title);
- 对高频关系查询建立合适的模式或限制遍历深度以控制查询成本。
- 对关系的权重或时间属性做 TTL/归档策略，避免历史关系无限增长影响性能。

数据导入与同步：

- 资源与知识点来自关系型数据库/文本解析，建议通过批量导入（neo4j-admin import）或使用安全的批处理脚本（CSV + Cypher LOAD CSV）定期同步。
- 对于实时更新，使用后端服务（FastAPI）以事务方式写入 Neo4j，避免直接在前端调用数据库。

## 向量数据库设计

### 环境说明

- 选型：Chroma（轻量易用，文件/数据库持久化）。
- 部署方式：以本地文件打开，生产环境建议持久化目录挂载到容器卷（如 /data/chroma）并做好定期备份。
- 存储与可用性：保证磁盘持久化、快照/备份；向量库文件可能较大，应监控磁盘使用。
- 连接接口：通过项目中的 Chroma 客户端或自定义 API 层访问向量集合，禁止直接从外部暴露磁盘路径。

### 数据结构

集合名：resources

每个向量项包含如下字段：

- id: string (主键，`<fileid>_chunk_<i>`)
- document: string (原始文本片段)
- embedding: vector[768] (浮点数组)
- metadata: JSON，包含`fileid` `filename` `filepath`

示例：

```json
{
  "id": "file123_chunk_3",
  "text": "二叉树的定义...",
  "embedding": [0.00123, -0.23, ...],
  "metadata": {"fileid": "xx", "filename": "file123", "filepath": "url"}
}
```
